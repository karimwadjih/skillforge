# Contributing to Skillforge

Skillforge accepts code, content, docs, benchmark, and tooling contributions. It does not accept the usual open-source shortcuts that make a repository look bigger than it is.

## Contribution principles

- Treat Skills like maintained workflow assets, not disposable prompts.
- Prefer sharper judgment over more volume.
- Keep public claims modest and technically justified.
- Label maturity honestly.
- Preserve portability and explicit contracts where possible.

## Before you open a pull request

1. Read [`AGENTS.md`](./AGENTS.md) and the certification standards in [`certification/`](./certification).
2. Check whether the Skill, collection, or benchmark asset already exists.
3. Regenerate and validate the repo locally.

```bash
pnpm skills:generate
pnpm score
pnpm catalog:generate
pnpm validate
pnpm test
```

If your change touches docs, packaging, or benchmark flows, run the relevant commands too.

## Adding or upgrading a Skill

1. Update `data/skill-seeds.json`.
2. Regenerate the library with `pnpm skills:generate`.
3. Improve the generated asset where the tier requires more depth.
4. Regenerate catalog artifacts with `pnpm catalog:generate`.
5. Run validation, tests, and any relevant packaging or benchmark steps.

## What good looks like

- The workflow is repeated and operationally meaningful.
- The trigger is specific enough to know when to reach for the Skill.
- Inputs and outputs are explicit.
- Examples look like real review or operating moments, not toy filler.
- Rubrics judge decision quality, not just writing quality.
- Edge cases and failure probes reveal actual workflow risk.
- The maturity label matches the evidence.

## What maintainers will reject

- Generic prompt-dump wording.
- Skills that sound like renamed versions of existing ones.
- Certification without benchmark coverage, score threshold, and manual review.
- Fake logos, endorsements, usage numbers, or unsupported compatibility claims.
- Contributions that increase breadth while weakening trust.

## Pull request expectations

- Keep the change scoped and reviewable.
- Explain whether the change affects seeded breadth, developed content, flagship content, or certification posture.
- Include benchmark or test updates when behavior changes.
- Call out any generated artifacts that must be regenerated by the reviewer.

## Maintainer policy

Maintainers may downgrade maturity labels, request stronger evidence, or reject contributions that feel autogenerated, redundant, or insufficiently differentiated.
