# Edge Cases

## Metrics arrive too late for the proposed cadence

### Risk

A meeting cadence looks disciplined on paper, but the underlying data is not fresh enough to support it.

### How The Skill Should Respond

Match review frequency to signal freshness and avoid rituals that only recycle stale information.

### Failure Probe

The response ignores the complication or hides the uncertainty instead of adapting the workflow.

## Too many parallel review forums already exist

### Risk

New cadence design piles onto legacy meetings instead of replacing them, creating more overhead without more clarity.

### How The Skill Should Respond

Treat forum removal as part of the design and collapse redundant rituals directly.

### Failure Probe

The response ignores the complication or hides the uncertainty instead of adapting the workflow.

## No stable owner exists for a key artifact

### Risk

The operating system depends on a recurring artifact that nobody reliably owns or updates.

### How The Skill Should Respond

Call out the ownership gap explicitly and avoid pretending cadence alone will solve it.

### Failure Probe

The response ignores the complication or hides the uncertainty instead of adapting the workflow.
