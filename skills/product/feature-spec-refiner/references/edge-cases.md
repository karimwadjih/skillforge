# Edge Cases

## Design is still fluid

### Risk

Key interaction details remain unsettled, which can make the spec sound more final than it really is.

### How The Skill Should Respond

Separate what is stable from what still depends on design resolution and avoid pretending the handoff is complete.

### Failure Probe

The response ignores the complication or hides the uncertainty instead of adapting the workflow.

## Hidden cross-team dependency

### Risk

A seemingly contained feature depends on another service, policy decision, or shared workflow that the spec barely mentions.

### How The Skill Should Respond

Pull the dependency into the main structure and show how it affects scope, timing, or acceptance criteria.

### Failure Probe

The response ignores the complication or hides the uncertainty instead of adapting the workflow.

## Stakeholders ask for wording cleanup only

### Risk

The team wants a cleaner document without surfacing the real ambiguity that will break implementation.

### How The Skill Should Respond

Treat clarity as a means to expose missing decisions, not as a cosmetic editing pass.

### Failure Probe

The response ignores the complication or hides the uncertainty instead of adapting the workflow.
