# Feature Spec Refiner

Tighten a draft spec until scope, state changes, acceptance criteria, and handoff risk are explicit.

## Quick View

- Category: `product`
- Maturity: `frontier`
- Tier: `developed`
- Evaluation status: `sampled`

## What This Skill Does

Specs fail less often because of typos than because they smuggle in ambiguous states, hidden dependencies, and acceptance criteria nobody could actually test.

## When To Reach For It

- A spec is nearly ready for handoff, but engineers or designers keep finding ambiguity in the details.
- The team has decided to build the feature, yet the document still hides assumptions that will surface during implementation.

## What You Should Provide

- draft feature spec
- goal or user problem
- constraints and non-goals
- dependencies across teams or systems
- open decisions or unresolved questions

## What You Should Get Back

- refined spec outline
- scope and assumptions gap list
- testable acceptance criteria
- handoff risks and follow-ups

## Repository Notes

- Main instructions live in `SKILL.md`.
- Machine-readable metadata lives in `skill.json`.
- Examples, rubric, edge cases, and tests live under `references/` and `tests/`.

## Quality Position

This Skill is meaningfully filled for v1, with stronger examples and tests than the broader seeded tier.
